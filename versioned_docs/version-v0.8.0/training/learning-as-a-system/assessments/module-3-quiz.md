# Module 3 Portfolio — Organizational Learning

**Level**: L3  
**Completion**: Evidence-based portfolio assessed by Team Lead + Principal Advisor  
**Prerequisite**: [Module 2 Quiz](./module-2-quiz.md) passed + ≥ 2 applied learning challenge ST analyses in GitHub issues  
**After completion**: Full LaaS certification; eligible to facilitate and mentor

Record completion in the [Progress Tracker](./progress-tracker.md).

---

## Overview

Module 3 is assessed through a **portfolio of practice evidence** — not a written quiz. The L3 practitioner demonstrates mastery by applying Learning as a System concepts at the organizational level across four evidence streams.

This portfolio checklist defines exactly what evidence is required, what quality means, and how to submit it.

---

## Required Evidence (All 4 items required)

### Evidence Item 1 — GitHub ST Analyses of Learning Challenges (≥ 2 issues)

**What**: Two or more GitHub issues (or issue comments) that apply a full ST Analysis to a real learning challenge — a situation where knowledge was expected but absent, skills degraded, a team repeatedly made the same mistakes, or where a training investment did not produce the expected behavioral change.

**Quality criteria**:
- Uses the canonical ST Analysis template from `docs/standards/issue-workflow.md`
- Affected stocks are correctly labeled (not flows mislabeled as stocks)
- At least one archetype named and correctly applied to the learning context
- Meadows leverage point cited with justification — not just a level number
- Structural fix proposed that addresses the root cause, not the symptom
- Analysis reflects L2 or L3 depth (not just L1 vocabulary)

**Submission format**:
```
Issue #1: [GitHub URL]
Summary: [One sentence — what was the learning failure? what was the structural root cause?]

Issue #2: [GitHub URL]
Summary: [One sentence]
```

---

### Evidence Item 2 — CLD in `docs/models/` (≥ 1)

**What**: One or more Causal Loop Diagrams of a team or personal learning system committed to `docs/models/issue-11/`. The CLD should go beyond Module 2 Exercise A — it should model an organizational or team-level learning system, not just a personal one.

**Quality criteria**:
- Uses Mermaid syntax that renders in the Docusaurus site
- ≥ 6 variables including at least one stock
- ≥ 2 loops (minimum one R, one B) with correct polarity
- At least one delay with a description of what the delay represents
- Reading Guide (3–5 sentences): what does the diagram reveal? what is the key leverage point and why?
- Models something real — a real team, process, or challenge (not a textbook example)

**Submission format**:
```
CLD file: docs/models/issue-11/[filename].md
Models: [what team/system/challenge?]
Loops: [list the loops by name — e.g., R1 Competence Loop, B1 Gap-Closing Loop]
Key leverage point: [variable + Meadows level]
```

---

### Evidence Item 3 — Facilitated Team Learning Retrospective (≥ 1)

**What**: One or more documented outputs from a facilitated "Our Learning System" retrospective session (see Module 3, Part 4 for the 60-minute session structure).

**Quality criteria**:
- Session involved ≥ 3 team members
- Documentation captures all three phases: current state map (stocks/flows identified), vulnerability identification, structural intervention proposals
- At least one reinforcing loop or balancing loop identified collaboratively
- Output document committed to `docs/models/issue-11/retros/` (or linked in a GitHub issue)
- Retrospective notes include: date, facilitator, participants (can be anonymized by role), and ≥ 1 action item

**Submission format**:
```
Retrospective doc: [GitHub URL or docs/ path]
Date: 
Participants: [count] — [roles, not names if privacy preferred]
Key insight: [One sentence — what structural element did the team identify that they hadn't seen before?]
Action item: [What structural change did the team commit to?]
```

---

### Evidence Item 4 — Peer Coaching Session on Learning System Design (≥ 1)

**What**: One or more documented peer coaching sessions in which you helped a colleague diagnose and redesign their personal or team learning system using the LaaS framework.

**Quality criteria**:
- Colleague was at L1 or L2 in their LaaS progression (you were coaching them forward)
- Session used ST vocabulary: stocks, flows, loops, delays, archetypes
- You identified at least one structural insight the colleague had not seen before
- You proposed at least one actionable structural change (specific flow to strengthen or outflow to reduce)
- Brief documentation committed to `docs/models/issue-11/coaching/` or added as a GitHub issue comment

**Submission format**:
```
Coaching doc/reference: [GitHub URL or docs/ path]
Colleague level: [L1 / L2]
Structural insight surfaced: [One sentence]
Structural change recommended: [One sentence — specifically what inflow/outflow/loop/delay to address]
```

## Portfolio Rubric

Team lead scores each evidence item on the criteria below. All items must reach at least **Proficient** on all criteria for L3 certification.

| Evidence Item | Criteria | Expert (4) | Proficient (3) | Developing (2) | Emerging (1) |
|---------------|----------|-----------|---------------|----------------|--------------|
| **GitHub Issues** (×2) | ST Analysis depth | Organizational-level loops, archetypal root cause, leverage point justified | Correct archetype + leverage point | Correct stocks/flows, loop identified | Vocabulary present, structural analysis incomplete |
| | Structural fix quality | Addresses root cause; identifies leverage point intervention | Addresses root cause | Symptomatic fix with acknowledgment | Symptomatic fix |
| **CLD** | Loop modeling | ≥ 3 loops, all polarities correct, delays described | ≥ 2 loops (R + B), correct polarities | 1 loop present, minor polarity errors | Variables mapped, no closed loop |
| | Insight quality | Reading guide reveals non-obvious organizational insight | Insight present, well stated | Insight thin but present | No reading guide or trivial observation |
| **Retrospective** | Facilitation quality | All phases completed; team identified at least 1 non-obvious loop; action item defined | All phases present; loop identified | Partial phases; vocabulary used but loop unclear | Event occurred but output thin |
| | Output documentation | Captures state map, vulnerabilities, interventions, action item | 3 of 4 elements present | 2 of 4 elements | 1 of 4 elements |
| **Peer Coaching** | ST depth of coaching | Coached colleague to identify their own structural insight | Surfaced structural insight; guided colleague | Shared L1/L2 concepts correctly | Shared vocabulary without structural application |
| | Actionability | Specific flow/loop intervention; colleague confirmed value | Specific intervention | General advice (e.g., "practice more") | No specific structural change |
