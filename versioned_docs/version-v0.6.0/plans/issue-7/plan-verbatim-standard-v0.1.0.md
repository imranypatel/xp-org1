## Problem Statement

Every plan document (`docs/plans/issue-N/...`) currently opens with front-matter metadata and then jumps directly to the Problem Statement. The verbatim CLI session text — the actual words that triggered the issue — lives only in the GitHub issue body, not in the plan itself.

This creates a traceability gap: a reader of the plan cannot see the original intent without navigating to GitHub. Including the verbatim text in the plan ensures the plan is self-contained and preserves the unfiltered signal that drove the decision to open the issue.

---

## 🔄 Systems Analysis

**Affected Stocks**: Traceability depth (ability to reconstruct original intent), Documentation self-containedness

**Affected Flows**:
- Inflow: Signal richness per plan (each plan now carries both raw intent and professional synthesis)
- Outflow: Intent drift (gap between what was asked and what was built)

**Feedback Loops**:
- **R1**: Better traceability → clearer decision history → better future decisions → more deliberate issue creation

**Delays**: None significant — this is a documentation-level change with immediate effect

**Archetype**: *Fixes that Fail* prevention — without verbatim capture, the Problem Statement becomes the only record, and future agents/contributors may optimize for the paraphrased intent rather than the original signal

**Leverage Point**: Meadows Level 5 — **Information flows**: preserving the original signal at the point of plan creation costs nothing but prevents drift across all future decisions

**ST Labels**: `st/leverage-point`

**Model**: Not modeled — structural simplicity of the change makes CLD unnecessary
